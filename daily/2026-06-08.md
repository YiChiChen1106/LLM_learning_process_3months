`DataCollatorWithPadding` 大约在第 191 行：

```python
@dataclass
class DataCollatorWithPadding:
    tokenizer: PreTrainedTokenizerBase
    padding: bool | str | PaddingStrategy = True
    max_length: int | None = None
    pad_to_multiple_of: int | None = None
    return_tensors: str = "pt"

    def __call__(self, features: list[dict[str, Any]]) -> dict[str, Any]:
        batch = pad_without_fast_tokenizer_warning(
            self.tokenizer,
            features,
            padding=self.padding,
            max_length=self.max_length,
            pad_to_multiple_of=self.pad_to_multiple_of,
            return_tensors=self.return_tensors,
        )
        if "label" in batch:
            batch["labels"] = batch["label"]
            del batch["label"]
        if "label_ids" in batch:
            batch["labels"] = batch["label_ids"]
            del batch["label_ids"]
        return batch
```

它的核心逻辑是：

```text
features: list[dict]
-> tokenizer padding
-> batch tensor dict
```

最后两段：

```python
if "label" in batch:
    batch["labels"] = batch["label"]
    del batch["label"]
if "label_ids" in batch:
    batch["labels"] = batch["label_ids"]
    del batch["label_ids"]
```

只是字段名统一：

```text
label     -> labels
label_ids -> labels
```

它不负责 assistant-only mask，也不负责把 padding labels 改成 `-100`。

## 为什么要看 DataCollatorForTokenClassification

接着看 `DataCollatorForTokenClassification`，因为它更接近我们关心的 labels padding。

关键字段：

```python
label_pad_token_id: int = -100
```

源码注释说明：

```text
padding labels 时用 -100；
-100 会被 PyTorch loss 忽略。
```

它的关键套路：

```python
label_name = "label" if "label" in features[0] else "labels"
labels = [feature[label_name] for feature in features] if label_name in features[0] else None

no_labels_features = [
    {k: v for k, v in feature.items() if k != label_name}
    for feature in features
]

batch = pad_without_fast_tokenizer_warning(
    self.tokenizer,
    no_labels_features,
    padding=self.padding,
    max_length=self.max_length,
    pad_to_multiple_of=self.pad_to_multiple_of,
    return_tensors="pt",
)
```

也就是：

```text
先把 labels 单独拿出来；
input_ids / attention_mask 交给 tokenizer.pad；
labels 自己手动补齐。
```

然后拿最终 batch 长度：

```python
sequence_length = batch["input_ids"].shape[1]
```

右 padding：

```python
batch[label_name] = [
    to_list(label) + [self.label_pad_token_id] * (sequence_length - len(label))
    for label in labels
]
```

左 padding：

```python
batch[label_name] = [
    [self.label_pad_token_id] * (sequence_length - len(label)) + to_list(label)
    for label in labels
]
```

这给我们的启发是：

```text
input_ids 的 padding 规则是补 pad_token_id；
labels 的 padding 规则是补 -100；
所以 labels 不应该当普通字段直接交给 tokenizer.pad。
```

## 右 padding 和左 padding

如果 tokenizer 是右 padding：

```python
label + [-100] * pad_len
```

例子：

```text
label = [20, 21]
pad_len = 3
-> [20, 21, -100, -100, -100]
```

如果 tokenizer 是左 padding：

```python
[-100] * pad_len + label
```

例子：

```text
label = [20, 21]
pad_len = 3
-> [-100, -100, -100, 20, 21]
```

labels 的 padding 方向必须和 `input_ids` 一致，否则 token 位置和 label 位置会错位。

## 动态 padding 改造思路

Day26 时当前脚本还是固定 padding：

```python
encoded = tokenizer(
    text,
    truncation=True,
    max_length=cfg["max_length"],
    padding="max_length",
)
```

今天改成动态 padding 后，`tokenize()` 阶段变成：

```python
encoded = tokenizer(
    text,
    truncation=True,
    max_length=cfg["max_length"],
    padding=False,
)
```

含义是：

```text
太长：仍然截断到 max_length
太短：先不补齐，交给 collator
```

`max_length` 仍然需要，用来限制单条样本最长不超过多少。

## 新增 SFTDataCollator

今天在 `projects/llm-lora-finetuning/train_lora.py` 新增了：

```python
class SFTDataCollator:
    def __init__(self, tokenizer, pad_to_multiple_of: int | None = None):
        self.tokenizer = tokenizer
        self.pad_to_multiple_of = pad_to_multiple_of

    def __call__(self, features: list[dict[str, list[int]]]) -> dict[str, torch.Tensor]:
        labels = [feature["labels"] for feature in features]
        no_labels_features = [
            {k: v for k, v in feature.items() if k != "labels"}
            for feature in features
        ]

        batch = self.tokenizer.pad(
            no_labels_features,
            padding=True,
            pad_to_multiple_of=self.pad_to_multiple_of,
            return_tensors="pt",
        )

        sequence_length = batch["input_ids"].shape[1]
        padded_labels = []
        for label in labels:
            label = label.tolist() if hasattr(label, "tolist") else list(label)
            pad_len = sequence_length - len(label)
            if self.tokenizer.padding_side == "right":
                padded_labels.append(label + [-100] * pad_len)
            else:
                padded_labels.append([-100] * pad_len + label)

        batch["labels"] = torch.tensor(padded_labels, dtype=torch.long)
        return batch
```

职责分工：

```text
tokenize:
生成不等长 input_ids / attention_mask / labels
其中 labels 已经做好 assistant-only mask

collator:
动态补齐 input_ids / attention_mask / labels
labels 新补出来的位置补 -100
```

## Trainer 传入 data_collator

今天也把 Trainer 初始化从：

```python
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized,
)
```

改成：

```python
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized,
    data_collator=data_collator,
)
```

其中：

```python
data_collator = SFTDataCollator(
    tokenizer,
    pad_to_multiple_of=cfg.get("pad_to_multiple_of"),
)
```

`pad_to_multiple_of` 是可选配置。如果配置里没有，默认就是 `None`。

## assistant-only mask 仍在 tokenize 阶段

动态 padding 不改变 assistant-only mask 的位置。

`tokenize()` 仍然负责：

```text
prompt / user / template -> -100
assistant response       -> token id
```

核心逻辑仍然是：

```python
encoded["labels"] = [
    token_id if mask == 1 and i >= prompt_len else -100
    for i, (token_id, mask) in enumerate(
        zip(encoded["input_ids"], encoded["attention_mask"])
    )
]
```

区别只是：

```text
以前 labels 已经补到 max_length；
现在 labels 是不等长的，等 collator 再补 -100。
```

## 边界情况：全 -100 样本

今天还讨论了一个边界情况：

```text
prompt 太长
-> 截断后 assistant response 一个 token 都没留下
-> labels 全是 -100
```

这种样本对 loss 没有贡献：

```text
占 batch 位置
参与 forward 计算
但没有有效训练信号
```

后续如果继续优化，可以考虑过滤这类样本。

今天先不实现过滤。

## 今天能回答的问题

### Q1: `DataCollatorWithPadding` 在哪里？

在：

```text
transformers\data\data_collator.py
```

它的核心是调用 tokenizer padding，把 `features: list[dict]` 变成 batch tensor dict。

### Q2: `label` / `label_ids` 改成 `labels` 是在做什么？

只是字段名统一，方便模型 forward 接收 `labels=...`。

### Q3: 为什么 labels 不能直接当普通字段交给 tokenizer.pad？

因为 `input_ids` padding 应该补 `pad_token_id`，而 `labels` padding 应该补 `-100`。

### Q4: 右 padding 时 labels 怎么补？

```python
label + [-100] * pad_len
```

### Q5: 左 padding 时 labels 怎么补？

```python
[-100] * pad_len + label
```

### Q6: 动态 padding 后 `max_length` 还需要吗？

需要。它仍然负责 truncation，防止单条样本过长。

### Q7: `padding=False` 表示什么？

表示 tokenize 阶段先不补齐，留给 collator 在 batch 阶段动态 padding。

## 今天的结论

```text
tokenize 阶段决定哪些 token 算 loss；
collator 阶段决定怎么把不等长样本补成 batch；
input_ids 补 pad_token_id；
attention_mask 补 0；
labels 补 -100。
```

再压缩成一句话：

```text
动态 padding 的关键不是少写 labels mask；
而是把“补齐长度”这件事从 tokenize 阶段移动到 collator 阶段。
```
