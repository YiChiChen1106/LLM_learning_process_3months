# 2026-06-15 - Day33 50 题评测集与 0.5B 边界

## 今天做了什么

今天没有继续盲目加训练集，而是把昨天的 `curated_eval` 从 12 题扩成了 50 题固定评测集，并把标注模板一一对齐。

新增的 50 题分成四类：

- `concept_*`：LoRA / QLoRA / target_modules / `lora_alpha` / `r`
- `dataflow_*`：dataset -> collator -> labels -> trainer -> generate
- `failure_*`：base/adapter 混用、target_modules 写错、`-100` 没处理好
- `boundary_*`：0.5B 能做什么、不能做什么

今天的核心不是“把模型训得更强”，而是把“能测、能判、能迭代”这条链路先固定下来。

## 0.5B 最适合什么

今天顺着这个问题重新想了一遍：`Qwen/Qwen2.5-0.5B-Instruct` 最适合的不是复杂推理，而是低成本实验和窄任务验证。

比较合适的场景是：

- 学习 LoRA / QLoRA 流程
- 跑通训练、保存、加载、评测
- 做模板化回答、改写、简单分类
- 低成本本地调试

它不适合拿来当“真正可靠的 LLM 老师”，因为它太小，复杂事实和推理都不稳。

## 为什么现在的数据像 toy，但仍然有价值

这个判断是对的。现在这套数据确实更像教学用 toy data，不像产品级语料。

但它的价值不在“让模型变强到可用”，而在“让我学会整条链路”：

1. 数据怎么进来
2. labels 怎么 mask
3. LoRA / QLoRA 怎么训练
4. adapter 怎么保存和加载
5. base 和 adapter 怎么对照评测
6. 错误怎么分类
7. 下一轮数据该怎么补

也就是说，toy data 的作用是把训练系统变成一个可观察的实验台。

## 今天跑出来的评测结果

我在远程 `pink-outer` 上跑了 base + 现有 `v6` adapter 的固定评测：

- `curated_eval.jsonl`: 50 题
- 输出: 100 条 response
- 保存到: `runs/day33_eval.jsonl` 和 `runs/day33_eval.md`

肉眼抽样能看到：

- base 模型大量乱编 LoRA / QLoRA 概念
- v6 比 base 短、更像在回答，但仍然有不少反向错误
- 典型问题是把 `base weights`、`adapter`、`quantization` 的关系说反

这说明：

- 有微调，不等于概念就真的对了
- 只看训练 loss 不够
- 固定评测集比继续堆训练数据更重要

## 今天的结论

今天最重要的一句话：

```text
0.5B 很适合拿来学流程，不适合拿来证明能力；toy 数据很像玩具，但它正好适合把训练闭环做扎实。
```

更具体一点：

- 先做固定评测，再补训练数据
- 先看错误类型，再决定下一轮怎么改
- 先把 base / adapter / eval 的关系弄清楚，再追更大的模型

## concept-fix 失败实验

后来又做了一轮小实验：

```text
用 10 条 concept_fix_train.jsonl 专门纠正 v6 的概念错误
训练一个新 adapter: qwen-concept-fix-lora
再用同一份 50 题 curated_eval 做 base / v6 / concept-fix 对比
```

训练本身跑通了：

```text
trainable params: 1,081,344
loaded examples: 10
kept examples: 10
train steps: 18
train loss: 3.196
```

但评测结果不理想。

前 10 条 concept 题人工标注：

```text
v6:          1 / 10 correct
concept-fix: 0 / 10 correct
```

更重要的是，`concept-fix` 的 hallucination 反而更多：

```text
v6 hallucinated_mechanism:          1
concept-fix hallucinated_mechanism: 6
```

这说明只拿 10 条纠错样本单独训练一个 adapter，不一定会让模型更懂概念。

它可能出现的问题是：

- 数据太少
- 问法太窄
- 训练目标太像“纠正句式”，不够像真实用户问题
- 小模型本身容易把 LoRA 编造成不存在的缩写
- 单独训练 concept-fix 可能破坏 v6 已经学到的简短回答风格

所以这轮失败也有价值：

```text
下一轮不应该只用 10 条纠错样本单独训。
更合理的是把基础概念样本 + 纠错样本混合起来，
并增加更多改写问法，再用同一份 curated_eval 验证。
```

## 下一步

下一步最自然的是：

1. 继续用 50 题固定评测集看输出
2. 把错误按类型归类
3. 只补错题对应的数据
4. 再训练下一轮 adapter

今天先不追更大的模型。

```text
curated_train + concept_fix_train -> mixed_train
再训练 mixed adapter
再跑同一份 50 题 eval
```

如果 mixed adapter 还是不行，再考虑扩大数据量或换更大的模型。

## mixed adapter 继续失败

最后又做了一轮 mixed 实验：

```text
curated_train.jsonl: 12 条基础概念样本
concept_fix_train.jsonl: 10 条纠错样本
mixed_train.jsonl: 22 条混合样本
```

训练也正常完成：

```text
loaded examples: 22
kept examples: 22
train steps: 36
train loss: 2.796
```

然后用同一份 50 题 `curated_eval.jsonl` 对比：

```text
base
v6
concept-fix
mixed
```

前 10 条 concept 题人工标注结果：

```text
v6:          1 / 10 correct
concept-fix: 0 / 10 correct
mixed:       0 / 10 correct
```

mixed 没有解决核心问题。

它比 concept-fix 少了一些离谱幻觉，但仍然把关键关系说错：

- 把 LoRA 说成会选择部分 base 参数更新
- 把 optimizer 更新对象说成 base weights / skip connection weights
- 把 lora_alpha 解释成多个 alpha 参数
- 把 QLoRA 显存节省和 adapter 参数多少混在一起
- 没能稳定说出“QLoRA 量化 frozen base weights，adapter 仍训练”

所以今天又得到一个更具体的结论：

```text
小模型 + 极小数据下，简单混合纠错样本并不一定提高概念稳定性。
```

下一轮不应该继续加类似句子，而应该换数据形态：

- 多写真实问法，而不是“请纠正这句话”
- 每个核心概念至少 5-8 个不同问法
- response 更短、更统一，减少模型自由发挥空间
- 先只训练 3 个核心概念：LoRA 训练谁、QLoRA 量化谁、adapter 能不能单独推理
- 或者换更大的模型再比较同样数据是否有效

## core-concepts 仍然失败

最后又尝试了一个更像真实用户问法的数据集：

```text
core_concepts_train.jsonl
18 条
只覆盖 3 个核心概念：
1. LoRA 训练谁
2. QLoRA 量化谁
3. adapter 能不能单独推理
```

这次不再使用“请纠正这句话”的格式，而是改成更自然的问题：

```text
LoRA 微调时，原模型的大部分参数会被改掉吗？
QLoRA 主要是把哪部分权重压到 4bit？
只有 adapter_model.safetensors，能直接推理吗？
```

训练正常完成：

```text
loaded examples: 18
kept examples: 18
train steps: 30
train loss: 2.772
```

但评测仍然失败。

前 10 条 concept 题人工标注：

```text
v6:            1 / 10 correct
concept-fix:   0 / 10 correct
mixed:         0 / 10 correct
core-concepts: 0 / 10 correct
```

`core-concepts` 的错误更像是：

- 句式学到了“先说结论”
- 但概念关系仍然说反
- 会把 LoRA 说成预训练
- 会说主要更新 base weights
- 会说 QLoRA 省显存是因为 adapter 参数少
- 对 adapter checkpoint 的理解仍然不稳

这说明问题已经不只是“问法像不像真实用户”。

今天更真实的结论是：

```text
0.5B + 十几条 LoRA 教学样本，不足以稳定学会这些抽象概念。
```

继续在 0.5B 上用十几条数据反复微调，收益很低。

下一步更合理的选择有两个：

```text
方案 A：把 0.5B 继续当 pipeline/debug 模型，不再期待它概念稳定。
方案 B：换 1.5B 或 3B，再用同一份数据和同一份 eval 比较模型规模的影响。
```

今天更推荐方案 A 先收住：把训练、评测、标注、失败分析全部记录下来。
