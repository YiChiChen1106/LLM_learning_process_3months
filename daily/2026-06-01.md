## Causal LM 为什么要 shift

假设 token 序列是：

```text
input_ids = [101, 200, 300, 400]
labels    = [101, 200, 300, 400]
```

表面看输入和答案一样，但 Causal LM 内部算 loss 时会做：

```python
shift_logits = logits[:, :-1, :]
shift_labels = labels[:, 1:]
```

所以真正的训练目标是：

```text
位置 0 的输出 -> 预测 200
位置 1 的输出 -> 预测 300
位置 2 的输出 -> 预测 400
```

最后一个位置没有下一个 token 可以预测，所以不参与 loss。

一句话记忆：

```text
input_ids 是模型读进去的前文；
labels 是模型要预测的答案；
Causal LM 内部通过 shift 让第 t 个位置预测第 t+1 个 token。
```

## shape 怎么对齐

如果：

```text
logits shape = [2, 4, 50000]
labels shape = [2, 4]
```

shift 之后：

```text
shift_logits shape = [2, 3, 50000]
shift_labels shape = [2, 3]
```

其中 `50000` 是 `vocab_size`，表示每个位置对整个词表中所有 token 的预测分数。

算 `CrossEntropyLoss` 前，通常会 flatten：

```text
shift_logits: [2, 3, 50000] -> [6, 50000]
shift_labels: [2, 3]        -> [6]
```

也就是 6 个分类任务，每个任务从 50000 个 token 里选正确答案。

## `-100` 的作用

PyTorch 的 `CrossEntropyLoss` 默认：

```python
ignore_index = -100
```

所以 label 为 `-100` 的位置会被忽略，不参与 loss。

它不是因为数值小才影响小，而是 loss 函数明确跳过它。

## padding label 为什么要 mask

昨天源码里原本是：

```python
encoded["labels"] = encoded["input_ids"].copy()
```

如果 tokenizer 返回：

```text
input_ids      = [101, 200, 300, 0, 0]
attention_mask = [1,   1,   1,   0, 0]
```

那么原始 labels 会是：

```text
[101, 200, 300, 0, 0]
```

这会让 padding token 也可能参与 loss。更理想的是：

```text
[101, 200, 300, -100, -100]
```

这样 shift 后：

```text
shift_labels = [200, 300, -100, -100]
```

真正训练的是：

```text
位置 0 -> 预测 200
位置 1 -> 预测 300
位置 2 -> 忽略
位置 3 -> 忽略
```

## 今天的代码修改

修改了 `projects/llm-lora-finetuning/train_lora.py` 里 `tokenize()` 的 label 生成逻辑：

```python
encoded["labels"] = [
    token_id if mask == 1 else -100
    for token_id, mask in zip(encoded["input_ids"], encoded["attention_mask"])
]
```

这个修改只影响哪些位置参与 loss：

```text
input_ids 不变
attention_mask 不变
labels 会变
```

它不改变 LoRA adapter 结构，也不会改变 trainable parameters 数量。

## 今天能回答的问题

### Q1: `labels=input_ids.copy()` 是不是让模型预测当前 token？

不是。Causal LM 内部会 shift，所以第 t 个位置预测第 t+1 个 token。

### Q2: 为什么第一个 label 不需要手动改成 `-100`？

因为 `shift_labels = labels[:, 1:]` 会把第一个 label 丢掉，它本来就不参与 loss。

### Q3: padding 位置为什么要改成 `-100`？

因为 padding 不是要学习的真实文本目标。改成 `-100` 后，`CrossEntropyLoss` 会忽略这些位置。

### Q4: 这个修改会改变 LoRA 参数量吗？

不会。它只改变 loss 计算的位置，不改变模型结构、rank、target modules 或 adapter 参数量。

## 今天的结论

```text
Causal LM 通过 shift 训练“预测下一个 token”；
padding 位置不是训练目标，应该在 labels 里标成 -100；
这个修复属于数据 label 处理，不属于模型结构修改。
```
