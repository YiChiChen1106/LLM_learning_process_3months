## Toy 到 PEFT 的映射

### 1. `qweight + scale` -> `load_in_4bit`

toy `QuantizedLinear` 里：

```text
qweight + scale
```

对应真实 QLoRA 里的量化 base weights。Hugging Face 里通过 `BitsAndBytesConfig(load_in_4bit=True, ...)` 配置量化加载。

### 2. `lora_A / lora_B` -> PEFT adapter

toy `QLoRALinear` 里：

```text
lora_A.weight
lora_B.weight
```

对应 PEFT 注入到目标 Linear 上的 LoRA adapter。它们是可训练参数。

### 3. `target_modules` -> 选择哪些 Linear 加 adapter

toy 里我们手动把一个 Linear 包成 `QLoRALinear`。PEFT 里用：

```python
target_modules=["q_proj", "k_proj", "v_proj", "o_proj"]
```

表示只给这些 attention projection 层加 LoRA adapter。

### 4. `r` -> `lora_rank`

toy 里的 rank：

```python
r=8
```

对应配置里的：

```yaml
lora_rank: 8
```

它决定中间瓶颈维度，也决定 adapter 参数量。

### 5. `alpha / r` -> `lora_alpha / r`

toy 里的：

```python
scaling = alpha / r
```

对应 PEFT 里的 `lora_alpha`。它控制 LoRA 分支加回 base output 时的缩放强度。

### 6. `adapter checkpoint` -> `save_pretrained` / `save_model`

toy 里 adapter checkpoint 只保存：

```text
lora_A.weight
lora_B.weight
```

PEFT 训练后保存的重点也是 adapter，而不是把完整 base model 一起塞进 adapter checkpoint。adapter 后续仍然要搭配同一个 base model 使用，或者 merge 后再用于独立推理。

## 今天能回答的问题

### Q1: adapter checkpoint 为什么不能单独 generate？

因为 adapter checkpoint 只有 LoRA 补丁参数，没有完整 base model、embedding、Transformer blocks、LayerNorm、lm_head、tokenizer 和 config。

### Q2: `target_modules` 选错会怎样？

如果名字选错，adapter 可能没有接到预期 Linear 上；如果选得太少，训练能力可能不足；如果选得太多，trainable parameters 和显存开销会增加。

### Q3: `r` 和 `lora_alpha` 分别控制什么？

`r` 控制 LoRA 的低秩瓶颈大小和 adapter 参数量。`lora_alpha` 控制 LoRA 分支输出加回 base output 时的缩放强度。

### Q4: QLoRA 为什么 trainable parameters 和 LoRA 基本一样？

因为 LoRA 和 QLoRA 通常训练的是同一组 adapter 参数。QLoRA 额外改变的是 frozen base weights 的存储精度，不是 adapter 的参数数量。

## 今天的一个误区

`load_in_4bit=True` 不等于把所有参数都变成 4-bit 训练。它主要是量化加载 frozen base model；LoRA adapter 仍然保持可训练的较高精度参数。

## 今天的结论

可以把真实工具里的 QLoRA 记成：

```text
BitsAndBytesConfig 负责量化 base model；
LoraConfig 负责定义 adapter 怎么接；
target_modules 负责决定 adapter 接到哪些 Linear；
save_pretrained / save_model 保存的是 adapter 结果。
```
