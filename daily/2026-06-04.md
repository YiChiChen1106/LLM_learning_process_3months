## collator 是什么

训练数据一开始是一条一条的样本：

```python
{
    "input_ids": [...],
    "attention_mask": [...],
    "labels": [...],
}
```

但模型训练需要的是 batch：

```python
{
    "input_ids": tensor([...]),
    "attention_mask": tensor([...]),
    "labels": tensor([...]),
}
```

所以 collator 的核心作用是：

```text
list of samples -> batch tensor dict
```

也就是：

```text
Dataset 单条样本
-> DataLoader 取出多条
-> collator 组 batch
-> Trainer / model
```

## 当前脚本为什么 collator 很简单

当前 `projects/llm-lora-finetuning/train_lora.py` 里 tokenize 使用：

```python
encoded = tokenizer(
    text,
    truncation=True,
    max_length=cfg["max_length"],
    padding="max_length",
)
```

这表示每条样本在 `dataset.map(tokenize)` 阶段就已经补到固定长度：

```text
input_ids      长度 max_length
attention_mask 长度 max_length
labels         长度 max_length
```

所以默认 collator 不需要决定怎么 padding，只需要把等长样本堆起来：

```text
[512]
[512]
[512]
-> [batch_size, 512]
```

这叫固定 padding。

## 固定 padding 和动态 padding

固定 padding：

```text
每条样本都补到全局 max_length
```

动态 padding：

```text
每个 batch 内，补到这个 batch 的最长样本
```

例子：

```text
max_length = 512

当前 batch 真实长度:
100, 120, 80, 110
```

固定 padding：

```text
512, 512, 512, 512
总 token 数 = 512 * 4 = 2048
```

动态 padding：

```text
120, 120, 120, 120
总 token 数 = 120 * 4 = 480
```

动态 padding 通常更省显存和计算，但 batch 之间 shape 会变化。

## 动态 padding 时补什么

动态 padding 时：

```text
input_ids      补 pad_token_id
attention_mask 补 0
labels         补 -100
```

原因是：

```text
attention_mask = 0
-> 告诉模型 attention 时别看 padding 位置

labels = -100
-> 告诉 loss 这个位置不要算训练目标
```

所以它延续了 Day25 的结论：

```text
attention_mask 管输入侧；
labels=-100 管监督信号侧。
```

## assistant-only mask 放在哪里

如果以后把 `train_lora.py` 改成动态 padding，比较简单的职责分工是：

```text
tokenize 阶段：
决定哪些 token 是训练目标
prompt / user / template -> -100
assistant response       -> token id

collator 阶段：
只负责把不等长样本补成 batch
input_ids      -> pad_token_id
attention_mask -> 0
labels         -> -100
```

也就是：

```text
tokenize 管“哪些 token 算 loss”
collator 管“怎么补成 batch”
```

## 三种 collator

### 默认 collator

适合样本已经等长：

```text
[512], [512], [512]
```

主要负责堆成 tensor：

```text
[batch_size, 512]
```

当前脚本最接近这种情况。

### DataCollatorWithPadding

适合样本还没 padding：

```text
[5], [8], [3]
```

它会动态 padding 到当前 batch 的最长长度：

```text
[8], [8], [8]
```

### 自定义 SFT collator

适合更复杂需求：

```text
动态 padding labels 补 -100
多轮对话只训练 assistant
packing 多个样本进一个 block
保留样本边界
```

今天只理解概念，不改训练脚本。

## pad_to_multiple_of

动态 padding 不一定只补到最长长度，有时会补到某个倍数：

```python
pad_to_multiple_of=8
```

比如当前 batch 最长是 121：

```text
121 -> 128
```

这样稍微多补一点 padding，但 tensor shape 对 GPU 更友好。

常见折中：

```text
不补到全局 max_length
但补到当前 batch 最长长度的 8/16 倍数
```

## length grouping

如果样本长度差异很大：

```text
20, 30, 400, 500
```

动态 padding 仍然会被最长样本拖高：

```text
全部补到 500
```

length grouping 的目标是：

```text
让同一个 batch 里的样本长度更接近
```

比如：

```text
batch1: 20, 30   -> 补到 30
batch2: 400, 500 -> 补到 500
```

这样动态 padding 会更有效。

## packing

packing 更进一步，把多个短样本拼进一个固定长度 block：

```text
[样本1][样本2][样本3]...
```

主要目的：

```text
减少 padding 浪费，提高 token 利用率
```

但 packing 会让 mask 更复杂，因为需要处理：

```text
样本边界
EOS
assistant-only mask
跨样本预测是否允许
labels 的 -100 怎么跨样本拼接
```

所以现在更合理的路线是：

```text
先理解固定 padding
再理解动态 padding
再理解 length grouping
最后再看 packing
```

今天不实现 packing。

## label smoother

今天还简单提到 `label_smoother`。

普通路径：

```text
Trainer -> model(**inputs) -> model forward 返回 loss
```

如果使用 label smoothing，Trainer 可能会拿到 `outputs + labels` 后自己算 loss。

label smoothing 的直觉是：

```text
不要让模型对正确答案过度自信
给其他 token 留一点点概率
```

当前 `train_lora.py` 没有显式开启 label smoothing，所以今天只知道它存在，不深入实现。

## toy collator 实验

今天跑了一个临时 Python snippet，不修改仓库文件。

两条样本：

```python
sample1 = {
    "input_ids": [10, 11, 12],
    "attention_mask": [1, 1, 1],
    "labels": [-100, 20, 21],
}

sample2 = {
    "input_ids": [30, 31, 32, 33, 34],
    "attention_mask": [1, 1, 1, 1, 1],
    "labels": [-100, -100, 40, 41, 42],
}
```

动态 padding 的最长长度是：

```text
5
```

输出：

```text
input_ids =
tensor([[10, 11, 12,  0,  0],
        [30, 31, 32, 33, 34]])

attention_mask =
tensor([[1, 1, 1, 0, 0],
        [1, 1, 1, 1, 1]])

labels =
tensor([[-100,   20,   21, -100, -100],
        [-100, -100,   40,   41,   42]])
```

shape：

```text
input_ids shape      = (2, 5)
attention_mask shape = (2, 5)
labels shape         = (2, 5)
```

对比固定 padding：

```text
dynamic batch tokens = 10
fixed max_length=512 tokens = 1024
saved token slots = 1014
```

所以这个实验验证了：

```text
input_ids      补 pad_token_id，也就是 0
attention_mask 补 0
labels         补 -100
```

## 今天能回答的问题

### Q1: collator 的输入和输出是什么？

输入是多条样本组成的 list，输出是一个 batch tensor dict。

### Q2: 当前脚本为什么不需要复杂 collator？

因为 tokenize 阶段已经 `padding="max_length"`，每条样本都等长。

### Q3: 动态 padding 补到什么长度？

补到当前 batch 里最长样本的长度，或补到它的某个倍数。

### Q4: 动态 padding 时 labels 应该补什么？

补 `-100`，因为新补出来的位置不是训练目标。

### Q5: `pad_to_multiple_of=8` 时，最长长度 121 会补到多少？

补到 128。

### Q6: length grouping 的目的是什么？

让同一个 batch 里的样本长度更接近，减少 padding 浪费。

### Q7: 为什么不急着实现 packing？

因为 packing 会让样本边界、EOS、assistant-only mask 和 labels 拼接都更复杂。

## 今天的结论

```text
tokenize 决定哪些 token 是训练目标；
collator 决定怎么把多条样本补齐并堆成 batch。
```

再压缩成一句话：

```text
固定 padding 简单；
动态 padding 更省；
packing 更强但更复杂。
```
