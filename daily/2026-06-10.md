## Trainer 怎么拿 loss

今天先复盘 Day28 的核心：

```text
Trainer 拿到 batch
batch 里有 input_ids / attention_mask / labels
Trainer 调 model(**batch)
model 内部看到 labels，计算 loss
Trainer 从 outputs.loss 取 loss
```

也就是简化成：

```python
outputs = model(
    input_ids=input_ids,
    attention_mask=attention_mask,
    labels=labels,
)

loss = outputs.loss
```

如果 batch 里没有 `labels`，causal LM 通常不会自动返回 `outputs.loss`。

没有 labels 时，模型通常只返回：

```text
logits
past_key_values
hidden_states 等
```

因为模型不知道你想在哪些位置算 loss，也不知道哪些位置要忽略。

## backward / step / zero_grad

训练循环里三个关键动作：

```python
loss.backward()
optimizer.step()
optimizer.zero_grad()
```

它们的职责不同。

`loss.backward()`：

```text
根据 loss 计算可训练参数的梯度；
把结果放到 param.grad；
不会直接更新参数。
```

`optimizer.step()`：

```text
根据 param.grad 更新参数；
真正让模型参数发生变化。
```

`optimizer.zero_grad()`：

```text
清空旧梯度；
避免下一次 backward 时把旧梯度继续累加进去。
```

可以记成：

```text
backward 算方向；
optimizer.step 按方向走一步；
zero_grad 清掉旧方向。
```

## 为什么要 zero_grad

PyTorch 里梯度默认会累加：

```text
param.grad = old_grad + new_grad
```

如果不清空梯度，下一次 backward 会把新梯度加到旧梯度上。

普通训练循环可以写成：

```python
optimizer.zero_grad()
loss.backward()
optimizer.step()
```

也可以写成：

```python
loss.backward()
optimizer.step()
optimizer.zero_grad()
```

核心是：每次真正更新参数后，要清掉上一轮梯度。

## batch size 是什么

batch size 可以先理解成：

```text
一次喂给模型多少条样本。
```

如果有 100 条训练数据：

```text
batch_size = 1
```

模型一次看 1 条：

```text
第 1 次：样本 1
第 2 次：样本 2
第 3 次：样本 3
...
```

如果：

```text
batch_size = 4
```

模型一次看 4 条：

```text
第 1 次：样本 1,2,3,4
第 2 次：样本 5,6,7,8
第 3 次：样本 9,10,11,12
...
```

batch size 越大：

```text
一次更新参考的样本更多；
梯度通常更稳定；
但显存占用更高。
```

显存增加的一个重要原因是 activations。

训练时 forward 会保存中间激活值，backward 要用它们算梯度。

batch 越大，同时保存的 activations 越多。

## gradient accumulation

如果显存放不下大 batch，可以用 gradient accumulation。

比如：

```yaml
per_device_train_batch_size: 1
gradient_accumulation_steps: 8
```

意思是：

```text
每次 GPU 只处理 1 条样本；
连续处理 8 个 micro batch；
累积 8 次梯度；
第 8 次后 optimizer.step() 更新一次参数。
```

这接近 effective batch size = 8，但显存接近每次 batch size = 1。

代价是训练更慢，因为要跑 8 次 forward/backward 才更新一次参数。

公式：

```text
effective batch size =
per_device_train_batch_size
* gradient_accumulation_steps
* num_gpus
```

今天先按单卡理解：

```text
effective batch size =
per_device_train_batch_size
* gradient_accumulation_steps
```

例子：

```yaml
per_device_train_batch_size: 2
gradient_accumulation_steps: 4
```

单卡 effective batch size：

```text
2 * 4 = 8
```

## 为什么 accumulation 时要除 loss

假设有 4 个 micro batch，每个梯度是：

```text
g1, g2, g3, g4
```

如果不除以 4，梯度累加后是：

```text
g_total = g1 + g2 + g3 + g4
```

这会比平均梯度大约放大 4 倍。

我们通常想模拟的是大 batch 的平均梯度：

```text
g_avg = (g1 + g2 + g3 + g4) / 4
```

所以 accumulation 时常见做法是：

```python
loss = loss / gradient_accumulation_steps
loss.backward()
```

因为梯度和 loss 成正比，所以每次 backward 得到的是：

```text
g1/4, g2/4, g3/4, g4/4
```

累积后：

```text
g1/4 + g2/4 + g3/4 + g4/4
= (g1 + g2 + g3 + g4) / 4
```

一句话：

```text
不除：累积的是梯度总和；
除了：累积的是梯度平均。
```

## micro step 和 global step

今天区分了两个概念：

```text
micro step:
一次小 batch 的 forward/backward。

global step:
一次 optimizer.step，也就是模型参数真正更新一次。
```

如果：

```yaml
gradient_accumulation_steps: 4
```

那么：

```text
4 个 micro steps = 1 个 global step
```

Trainer 日志里的 `global_step` 通常更接近 optimizer step 次数，而不是 micro batch forward 次数。

## logging_steps 和 save_steps

`logging_steps` 和 `save_steps` 通常按 global step 计数。

如果：

```yaml
gradient_accumulation_steps: 8
logging_steps: 1
save_steps: 20
```

那么：

```text
1 global step = 8 micro steps
```

所以：

```text
logging_steps = 1
=> 每 1 个 global step 打一次日志
=> 大约每 8 个 micro steps 打一次日志
```

```text
save_steps = 20
=> 每 20 个 global steps 保存一次 checkpoint
=> 大约每 160 个 micro steps 保存一次 checkpoint
```

## toy training loop snippet

今天跑了一个临时 snippet，不改仓库文件：

```python
num_samples = 80
micro_batch_size = 2
gradient_accumulation_steps = 4
logging_steps = 1
save_steps = 3
```

计算结果：

```text
effective_batch_size = 8
num_micro_steps = 40
```

节奏是：

```text
micro_step 01: backward，只累积
micro_step 02: backward，只累积
micro_step 03: backward，只累积
micro_step 04: backward，然后 optimizer.step -> global_step 1
```

所以：

```text
4 个 micro steps = 1 个 global step
40 个 micro steps = 10 个 global steps
```

因为：

```text
save_steps = 3
```

checkpoint 出现在：

```text
global_step 3, 6, 9
```

换算成 micro step：

```text
micro_step 12, 24, 36
```

## 对照 smoke 配置

当前 `projects/llm-lora-finetuning/configs/smoke.yaml`：

```yaml
model_name: Qwen/Qwen2.5-0.5B-Instruct
train_file: data/sample_train.jsonl
output_dir: runs/qwen-smoke-lora
max_length: 512
pad_to_multiple_of: 8
per_device_train_batch_size: 1
gradient_accumulation_steps: 8
learning_rate: 0.0002
num_train_epochs: 1
logging_steps: 1
save_steps: 20
lora_rank: 8
lora_alpha: 16
lora_dropout: 0.05
load_in_4bit: false
target_modules:
  - q_proj
  - k_proj
  - v_proj
  - o_proj
```

## smoke 配置里的训练节奏

```yaml
per_device_train_batch_size: 1
```

单卡时，每个 micro step 真正塞进 GPU 的样本数是 1。

```yaml
gradient_accumulation_steps: 8
```

每 8 个 micro steps 才做一次：

```python
optimizer.step()
```

单卡 effective batch size：

```text
1 * 8 = 8
```

```yaml
logging_steps: 1
```

每 1 个 global step 打一次日志。

因为：

```text
1 global step = 8 micro steps
```

所以大约每 8 个 micro steps 打一次日志。

```yaml
save_steps: 20
```

每 20 个 global steps 保存一次 checkpoint。

换算成 micro steps：

```text
20 * 8 = 160 micro steps
```

```yaml
num_train_epochs: 1
```

整个训练集过 1 遍。

```yaml
learning_rate: 0.0002
```

每次 `optimizer.step()` 更新参数时的步长。

注意 learning rate 作用在 global step 上，不是每个 micro step 都更新参数。

## 80 条样本的例子

假设有效样本有 80 条：

```yaml
per_device_train_batch_size: 1
gradient_accumulation_steps: 8
num_train_epochs: 1
```

一个 epoch 有：

```text
80 / 1 = 80 micro steps
```

每 8 个 micro steps 更新一次：

```text
80 / 8 = 10 global steps
```

所以训练 1 epoch 大约有 10 次 optimizer step。

如果：

```yaml
save_steps: 20
```

中间不会触发 checkpoint，因为总共只有 10 个 global steps。

但脚本最后还有：

```python
trainer.save_model(cfg["output_dir"])
```

所以训练结束仍会保存最终 adapter/model 输出。

## smoke 配置里的长度参数

```yaml
max_length: 512
```

单条样本 tokenize 后最多 512 个 token。

太长会被截断。

```yaml
pad_to_multiple_of: 8
```

batch 阶段动态 padding 后，再把长度向上对齐到 8 的倍数。

它不是固定 padding 到 512。

例子：

```text
当前 batch 最大长度 = 5
pad_to_multiple_of = 8
最终长度 = 8

当前 batch 最大长度 = 9
pad_to_multiple_of = 8
最终长度 = 16
```

## smoke 配置里的 LoRA 参数

```yaml
lora_rank: 8
```

控制 LoRA 低秩矩阵的 rank。

如果一个 Linear 是：

```text
in_features -> out_features
```

LoRA 大概新增：

```text
A: in_features -> r
B: r -> out_features
```

新增参数量大约：

```text
r * (in_features + out_features)
```

`r` 越大，adapter 参数越多，表达能力更强，显存和计算也更多。

```yaml
lora_alpha: 16
```

控制 LoRA delta 的缩放。

常见：

```text
scaling = alpha / r
```

当前配置：

```text
alpha = 16
r = 8
scaling = 2
```

```yaml
lora_dropout: 0.05
```

训练 LoRA adapter 时对 LoRA 分支做一点 dropout。

它不是随机删除 5% 训练样本，而是训练时的一种正则化。

## target_modules

当前：

```yaml
target_modules:
  - q_proj
  - k_proj
  - v_proj
  - o_proj
```

表示给 attention 里的这些 Linear 加 LoRA adapter：

```text
q_proj: 生成 query
k_proj: 生成 key
v_proj: 生成 value
o_proj: attention 输出投影
```

不是 tokenizer。

tokenizer 只是把文本转成 token id，不是 LoRA adapter 插入的位置。

如果 `target_modules` 写错，比如写成 `q_projj`：

```text
PEFT 找不到目标模块；
LoRA 可能插不进去；
或者直接报错。
```

## load_in_4bit

当前：

```yaml
load_in_4bit: false
```

所以 smoke 是普通 LoRA，不是 QLoRA。

如果之后改成：

```yaml
load_in_4bit: true
```

脚本会走：

```python
quantization_config = BitsAndBytesConfig(...)
model = AutoModelForCausalLM.from_pretrained(
    ...,
    quantization_config=quantization_config,
)
model = prepare_model_for_kbit_training(model)
```

那才是 QLoRA 路线。

## 今天能回答的问题

### Q1: 没有 labels，模型会自动返回 loss 吗？

通常不会。

没有 labels 时，causal LM 通常只返回 logits 等输出。

### Q2: `loss.backward()` 会不会更新参数？

不会。

它只计算梯度，把结果放到 `param.grad`。

### Q3: 真正更新参数的是谁？

```python
optimizer.step()
```

### Q4: 为什么要 `optimizer.zero_grad()`？

因为 PyTorch 梯度默认累加。

不清空的话，下一次 backward 会把旧梯度加进去。

### Q5: batch size 是什么？

一次喂给模型多少条样本。

### Q6: effective batch size 怎么算？

单卡时：

```text
effective batch size =
per_device_train_batch_size * gradient_accumulation_steps
```

### Q7: 显存不够时优先调小哪个？

优先调小：

```yaml
per_device_train_batch_size
```

因为它决定一次真的塞进 GPU 的样本数。

### Q8: `global_step` 通常是什么？

通常是 optimizer step 的次数，也就是模型参数真正更新的次数。

### Q9: `logging_steps` 和 `save_steps` 通常按什么计数？

通常按 global step 计数。

### Q10: 当前 smoke 是 LoRA 还是 QLoRA？

普通 LoRA。

因为：

```yaml
load_in_4bit: false
```

## 今天的结论

```text
micro step 是一次小 batch 的 forward/backward；
global step 是一次 optimizer.step；
gradient_accumulation_steps 决定几个 micro step 合成一个 global step；
logging_steps 和 save_steps 通常按 global step 算。
```

再压缩成一句话：

```text
看训练日志时，不要把 step 理解成样本条数；它通常指参数更新次数。
```
