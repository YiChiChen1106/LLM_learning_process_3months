# 2026-06-12 - Day31 个人风格 LoRA 多轮训练与固定评测

## 今天做了什么

- 在 `pink-outer` 上从 smoke 进入真实小数据 LoRA 训练。
- 用 `Qwen/Qwen2.5-0.5B-Instruct` 跑了个人风格数据集。
- 连续做了 V1 到 V6 六轮 adapter 实验。
- 跑过 LoRA，也跑过同数据 QLoRA 对照。
- 新增固定评测集 `eval_prompts.jsonl`。
- 新增 `evaluate_adapters.py`，可以把多轮 adapter 输出保存成 JSONL 和 Markdown。
- 给评测脚本加了轻量单元测试。
- 最后确认：继续盲目加小数据不如先固定评测、定位错误、再改数据。

## V1 到 V6 总览

今天最重要的不是“训练出了一个好模型”，而是第一次完整体验了：

```text
造数据
-> 训练 adapter
-> 加载 base + adapter 生成
-> 发现错误
-> 改数据
-> 再训练
-> 固定评测对比
```

这比只跑一次 smoke 更接近真实微调工作流。

## V1：先让模型学“学习搭子风格”

V1 数据：

```text
personal_style_train.jsonl
18 条
```

目标很简单：让模型学会更像今天的学习节奏。

典型风格是：

```text
先说结论：...
然后用很短的解释补充。
```

训练结果：

```text
trainable params: 1,081,344
all params: 495,114,112
trainable%: 0.2184
adapter_model.safetensors: 约 4.2MB
```

V1 的现象：

- 学到了一点“先说结论”的格式。
- 但事实很不稳。
- 会把 LoRA 说成预训练模型。
- 会把 adapter checkpoint 解释成不完整甚至奇怪的东西。

这说明小数据很容易先学到表面格式，但不一定学稳知识。

## QLoRA 对照

同一份 V1 数据也跑了 QLoRA。

只改：

```yaml
load_in_4bit: true
```

关键观察：

```text
LoRA trainable params:  1,081,344
QLoRA trainable params: 1,081,344
```

这验证了前面学过的结论：

```text
QLoRA 省的是 frozen base weights 的显存，
不是减少 LoRA adapter 的可训练参数量。
```

QLoRA adapter 文件大小也约 4.2MB，和 LoRA adapter 差不多。

推理时确认：

```text
is_loaded_in_4bit: True
```

也就是 4bit base + adapter 的路径确实跑通了。

## V2：增加事实纠错数据

V2 数据：

```text
personal_style_train_v2.jsonl
37 条
```

目标是修正 V1 的事实错误：

- LoRA 不是通信技术。
- LoRA 不是预训练模型。
- adapter checkpoint 不是完整模型。
- QLoRA 量化的是 frozen base weights。

训练结果：

```text
trainable params: 1,081,344
train_loss: 2.707
epoch: 5
```

V2 的效果：

- 对“LoRA 是 Low Rate Communication 吗？”能更好地否定。
- 对“LoRA 是预训练模型吗？”也有改善。
- 但仍然会在长回答后半段混入错误概念。

结论：

```text
增加事实数据有效，但模型仍会概念混合。
```

## V3：hard negative 纠错

V3 数据：

```text
personal_style_train_v3.jsonl
38 条
```

这轮不是泛泛解释，而是专门针对模型犯过的错写 hard negative。

重点纠正：

- LoRA 不是预训练方法。
- LoRA 不会把大模型变成小模型。
- adapter checkpoint 不是预训练模型。
- adapter_model.safetensors 不是完整模型。
- QLoRA 不量化 adapter。

训练结果：

```text
train_loss: 1.67
epoch: 8
```

V3 的效果：

- `adapter checkpoint 为什么不能单独 generate？` 这题明显变好。
- 它能说出 adapter checkpoint 不是完整模型，需要 base model。
- 但又出现了新问题：

```text
base_model.ckpt
Qwen 说的是...
```

这类不存在或不可靠的机制开始冒出来。

结论：

```text
hard negative 能压住一些核心错误，
但长回答仍然容易乱补机制。
```

## V4：短回答、保守回答

V4 数据：

```text
personal_style_train_v4.jsonl
49 条
```

目标是让模型少说一点，避免后半段乱补。

策略：

- response 变短。
- 每条只保留 2 到 3 句。
- 推理时 `max_new_tokens` 也降到 56。

训练结果：

```text
train_loss: 1.715
epoch: 8
```

V4 的问题：

- 回答确实变短。
- 但短不等于正确。
- 仍然出现：

```text
低纬度参数迁移
卷积权重
adapter_model.safetensors 包含 base model
QLoRA quantize LoRA adapter
```

结论：

```text
只训练“短”不够。
短回答如果事实错，仍然是错。
```

## 固定评测集

V1 到 V4 之后，开始意识到不能只靠肉眼随便问几个 prompt。

新增固定评测集：

```text
projects/llm-lora-finetuning/eval_prompts.jsonl
```

里面固定了 10 个高风险问题：

- LoRA 是不是预训练方法。
- LoRA 会不会把大模型变小。
- adapter checkpoint 为什么不能单独 generate。
- 只给 adapter_model.safetensors 能不能推理。
- QLoRA 会不会量化 adapter。
- QLoRA trainable params 为什么和 LoRA 一样。
- target_modules 是什么。
- labels 里的 -100 是什么。
- 用保守方式解释 adapter checkpoint。

新增评测脚本：

```text
projects/llm-lora-finetuning/evaluate_adapters.py
```

它可以一次评测多轮 adapter：

```bash
python evaluate_adapters.py \
  --model Qwen/Qwen2.5-0.5B-Instruct \
  --prompts eval_prompts.jsonl \
  --adapter lora-v4=runs/qwen-personal-style-lora-v4 \
  --adapter lora-v5=runs/qwen-personal-style-lora-v5 \
  --output-jsonl runs/personal-style-eval-v4-v5.jsonl \
  --output-md runs/personal-style-eval-v4-v5.md \
  --max-new-tokens 56
```

输出有两份：

```text
JSONL: 方便程序扫错
Markdown: 方便人工读
```

单元测试：

```text
projects/llm-lora-finetuning/test_evaluate_adapters.py
```

验证结果：

```text
3 passed
```

## V5：只针对固定评测错误改数据

V5 数据：

```text
personal_style_train_v5.jsonl
48 条
```

这轮不再泛泛加数据，而是针对 V4 固定评测暴露的问题：

- 不要说卷积权重。
- 不要说 adapter_model.safetensors 包含 base。
- 不要说需要转换成 base_model.ckpt。
- 不要说 QLoRA 量化 adapter。

训练结果：

```text
train_loss: 1.631
epoch: 8
```

V5 的进步：

- 修掉了一些明显坏词。
- 不再出现卷积。
- 不再出现 base_model.ckpt。
- adapter checkpoint 不能单独 generate 这类问题更稳。

但 V5 也出现新错误：

```text
LoRA 是低秩 adapter 微调方法。它不是预训练方法，也不是微调方法。
```

这句话自相矛盾。

还出现：

```text
q_proj: question key
-100 是模型初始状态
adapter checkpoint 不影响完整模型的推理
```

结论：

```text
固定评测集起作用了。
它能告诉我们旧错误修了没有，也能暴露新错误。
```

## V6：针对 V5 新错误继续修

V6 数据：

```text
personal_style_train_v6.jsonl
36 条
```

这轮针对 V5 新错误：

- LoRA 是微调方法，不是“不是微调方法”。
- q_proj 是 query projection，不是 question key。
- labels 里的 -100 是 loss ignore index，不是模型状态。
- adapter checkpoint 会影响推理输出，不是“不影响完整模型”。
- QLoRA trainable params 来自 adapter，不是 base weights。

训练结果：

```text
train_loss: 2.01
epoch: 6
```

V6 的结果反而变差：

- 把 LoRA 说成预训练模块。
- 把 LoRA 说成 Low-Rate Update。
- 说 LoRA 会去掉不相关参数。
- 说 QLoRA 训练时只更新 base weights。
- 说 adapter checkpoint 包含 base model 的参数和 state_dict。

这说明继续用少量人工改写数据追着错误修，可能会让模型在概念空间里漂移。

## 今天的核心结论

今天真正学到的是：

```text
小数据 LoRA 可以快速改变风格，
但不保证事实稳定。
```

更具体地说：

```text
1. 风格比事实更容易被小数据学到。
2. loss 下降不等于答案事实正确。
3. hard negative 能修局部错误，但也可能引入新混淆。
4. 固定评测集比随便问几个 prompt 更可靠。
5. 继续盲目加几十条数据，不如先做系统评测和错误归因。
```

## 下一步怎么做

今天不建议继续 V7。

更合理的下一步是：

```text
1. 保留固定评测集。
2. 把错误项分成事实错误、格式错误、幻觉扩展三类。
3. 减少训练轮次，避免小数据过拟合后漂移。
4. 使用更高质量、更一致的数据。
5. 后面再考虑更强 base model，例如 Qwen 1.5B / 3B / 7B。
```

一句话总结：

```text
今天从“能训练 adapter”进阶到了“知道训练出来的 adapter 为什么不可靠，以及怎么评测它”。
```
