## 为什么要过滤全 -100 labels

SFT 训练里，`labels=-100` 表示这个位置不参与 loss。

常见三类 token：

```text
prompt token:
attention_mask = 1
labels = -100

response token:
attention_mask = 1
labels = token_id

padding token:
attention_mask = 0
labels = -100
```

如果一条样本最后变成：

```text
labels = [-100, -100, -100]
```

它不会给 loss 提供任何有效训练信号。

这种情况常见于：

```text
prompt 太长
max_length 太短
assistant response 被截断没了
```

保留这种样本的问题是：

```text
它会占 batch 位置；
它会参与 forward 计算；
但它不贡献 loss。
```

所以今天新增过滤逻辑。

## 新增 has_trainable_label

今天在 `train_lora.py` 里新增：

```python
def has_trainable_label(example: dict[str, list[int]]) -> bool:
    return any(label != -100 for label in example["labels"][1:])
```

注意这里检查的是：

```python
example["labels"][1:]
```

不是完整的：

```python
example["labels"]
```

原因是 causal LM loss 会 shift labels。

## 过滤数据集

在 tokenize 之后：

```python
tokenized = dataset.map(tokenize, remove_columns=dataset.column_names)
tokenized = tokenized.filter(has_trainable_label)
```

如果过滤后没有样本，直接报错：

```python
if len(tokenized) == 0:
    raise ValueError("No trainable examples remain after filtering labels.")
```

这样比继续训练更好。继续训练只会在后面产生更难理解的问题。

## pad_to_multiple_of: 8

今天在 smoke 配置里加了：

```yaml
pad_to_multiple_of: 8
```

它的含义是：

```text
先按当前 batch 最大长度动态 padding；
再把长度向上对齐到 8 的倍数。
```

例子：

```text
当前 batch 最大长度 = 5
pad_to_multiple_of = 8
最终长度 = 8

当前 batch 最大长度 = 9
pad_to_multiple_of = 8
最终长度 = 16
```

它不是固定 padding 到 `max_length=512`，而是在动态 padding 的基础上做倍数对齐。

## labels 也必须 padding 到同样长度

如果 `input_ids` 被 padding 到长度 8：

```text
input_ids      -> 长度 8
attention_mask -> 长度 8
labels         -> 长度 8
```

labels 必须和 input_ids 对齐。

区别是补充值不同：

```text
input_ids      补 pad_token_id
attention_mask 补 0
labels         补 -100
```

## Trainer 怎么拿 loss

今天复盘了 Hugging Face Trainer 的训练路径。

核心不是 Trainer 自己手写 causal LM loss，而是：

```python
outputs = model(**inputs)
loss = outputs.loss
```

也就是说，batch 里的：

```text
input_ids
attention_mask
labels
```

会一起传给模型。

如果是 `Qwen2ForCausalLM` 这类 causal LM，模型 forward 看到 `labels` 后，会在模型内部计算 loss，然后返回 `outputs.loss`。

## Causal LM loss shift

causal LM 不是用当前位置预测当前位置，而是：

```text
位置 0 的 logits -> 预测位置 1 的 label
位置 1 的 logits -> 预测位置 2 的 label
位置 2 的 logits -> 预测位置 3 的 label
```

可以理解成：

```python
shift_labels = labels[:, 1:]
```

有些 Transformers 版本源码会写成：

```python
labels = pad(labels, (0, 1), value=-100)
shift_labels = labels[..., 1:]
loss = cross_entropy(logits, shift_labels, ignore_index=-100)
```

它右边补一个 `-100`，是为了让最后一个 logits 没有目标 token，因此不参与 loss。

## toy shift 例子

今天跑了一个临时 snippet：

```python
labels = [[-100, -100, 20, 21, -100]]
shift_labels = labels[:, 1:]
```

输出逻辑是：

```text
labels       = [[-100, -100, 20, 21, -100]]
shift_labels = [[-100, 20, 21, -100]]
active_mask  = [[False, True, True, False]]
targets used in loss = [20, 21]
```

对应关系：

```text
logits at original pos 0 -> label at original pos 1: ignored
logits at original pos 1 -> label at original pos 2: predict token 20
logits at original pos 2 -> label at original pos 3: predict token 21
logits at original pos 3 -> label at original pos 4: ignored
```

所以第一枚 response token 的 loss，来自它前一个位置的 logits。

## 为什么过滤规则要看 labels[1:]

如果只看完整 labels：

```python
labels = [20, -100, -100]
```

会误以为有训练信号。

但 causal LM shift 后：

```text
labels padded = [20, -100, -100, -100]
shift_labels  = [-100, -100, -100]
```

真正参与 loss 的 labels 全是 `-100`。

所以它没有训练信号。

因此今天把过滤规则定成：

```python
any(label != -100 for label in labels[1:])
```

而不是：

```python
any(label != -100 for label in labels)
```

## 今天能回答的问题

### Q1: `attention_mask` 和 `labels=-100` 的区别是什么？

`attention_mask` 管模型能不能看这个 token。

`labels=-100` 管这个位置算不算 loss。

### Q2: prompt token 应该怎么设置？

```text
attention_mask = 1
labels = -100
```

prompt 要给模型看，但不让它在 prompt 上算 loss。

### Q3: response token 应该怎么设置？

```text
attention_mask = 1
labels = token_id
```

response 是模型要学习预测的目标。

### Q4: padding token 应该怎么设置？

```text
attention_mask = 0
labels = -100
```

padding 既不应该被当成真实上下文，也不应该参与 loss。

### Q5: Trainer 是自己算 loss 吗？

通常不是。

Trainer 会把 batch 传给模型：

```python
outputs = model(**inputs)
loss = outputs.loss
```

causal LM 模型内部看到 `labels` 后计算 loss。

### Q6: 为什么 `labels[0]` 不重要？

因为 causal LM loss 会 shift。

位置 `0` 的 label 不会作为目标进入 loss，真正用于 loss 的是 shift 后的 `labels[1:]`。

### Q7: 为什么最后一个 logits 不参与 loss？

因为最后一个位置没有下一个 token 可预测。

源码会让它对应到 `-100`，所以被 `ignore_index=-100` 忽略。

## 今天的结论

```text
SFT 不是只要 input_ids 进模型就能学习；
真正决定有没有学习信号的是 shift 后的 labels 里有没有非 -100。
```

再压缩成一句话：

```text
过滤无效样本时，要按 causal LM 的 loss shift 规则检查 labels[1:]。
```
